CLAUDE.md, llms.txt, and spec.md: Why Developers Are Building Context Systems for AI

Every serious AI coding setup now has a rules file. Here is what each one does, where they break down, and what comes next.

Chris Hoar • February 2026 • 10 min read

I counted the context files in my current project last week. CLAUDE.md for Claude Code. A .cursorrules file for Cursor. An llms.txt for the website. A spec.md I paste into ChatGPT at the start of every session. Four different files, all trying to solve the same problem: telling AI tools what my project is and how I want them to behave.

This is not just me. Every developer I know who uses AI tools seriously has some version of this setup. We have independently converged on creating manual context files because AI tools do not retain project knowledge between sessions.

This is not a niche trick anymore. Martin Fowler's ThoughtWorks team published a full article on "Context Engineering for Coding Agents" in 2026. Addy Osmani calls it "context packing" and dedicates an entire section to it in his workflow guide. The Honeycomb blog has three posts on managing AI context. This is now a standard part of the AI-assisted development workflow.

The pattern has a name now: manual context systems. And they are everywhere.

What is CLAUDE.md and why does every Claude Code project have one?

CLAUDE.md is a markdown file in your project root that Claude Code reads at the start of every session. It contains project rules, coding conventions, architecture decisions, and behavioral instructions. Think of it as a persistent system prompt for your AI coding assistant. Claude Code loads it automatically, so you do not have to re-explain your project every time.

Here's what goes in a typical CLAUDE.md file. Project description and purpose. Tech stack details ("React 18.2 with TypeScript 5.3"). Coding style rules ("use async/await, never .then()"). File structure notes ("all API routes go in src/api/"). Things to avoid ("never modify the auth module without asking"). Testing conventions ("write Vitest unit tests for all utils").

Claude Code also supports a .claude/rules/ directory for path-scoped rules. You can have rules that only apply to *.ts files, or different rules for your frontend versus backend code. And Skills let you define reusable workflows that Claude can follow.

I use CLAUDE.md in every project now. It works well within Claude Code. But it does nothing when you switch to ChatGPT, Gemini, or Cursor with a non-Claude model. Your carefully crafted context stays locked in one tool.

What is .cursorrules and how is it different from CLAUDE.md?

.cursorrules (and the newer .cursor/rules/ directory) is Cursor's equivalent of CLAUDE.md. It tells Cursor's AI features how to behave in your project. The format is similar, markdown with instructions, but it only works inside the Cursor editor. If you use both Claude Code and Cursor, you need both files, often with overlapping content.

Cursor is moving toward a more structured rules system similar to Claude Code's path-scoped approach. You can now have different rules for different file types or directories. But again, this only works in Cursor.

There's also the emerging AGENTS.md standard that attempts to be tool-agnostic. GitHub Copilot started recognizing it recently. Windsurf supports it too. The idea is one context file that multiple tools can read. But adoption is still spotty, and each tool interprets it slightly differently.

The fragmentation is real. I have projects with CLAUDE.md, .cursorrules, and AGENTS.md all saying basically the same thing in slightly different formats.

What is llms.txt and should your project have one?

llms.txt is an emerging web standard, similar to robots.txt, that provides machine-readable context about a website or project for LLM crawlers. While CLAUDE.md and .cursorrules are for coding assistants, llms.txt is designed for any LLM that might need to understand your site or documentation. It is still experimental, but platforms like Perplexity have started honoring it.

The use case is different. If you maintain a library or API, an llms.txt file helps AI tools give better answers about your project. When someone asks ChatGPT "how do I use the FooBar library?", your llms.txt provides the context.

The format is structured text with sections like DESCRIPTION, DOCUMENTATION, API_ENDPOINTS, and EXAMPLES. It serves a different purpose than the coding context files. It is about discoverability, not development workflow.

I add llms.txt to public projects now. It takes five minutes and helps AI tools understand what you've built. But it does not solve the context problem for your development workflow.

What is spec.md and how do developers use it for AI coding?

spec.md is not a standard, but it has become a common convention. Developers write a detailed project specification in markdown and feed it to their AI tool at the start of a coding session. Addy Osmani describes creating a spec.md with requirements, architecture decisions, data models, and testing strategy before writing any code. It acts as the shared understanding between you and the AI.

This is the most manual of all the approaches. You write it, you maintain it, you copy-paste it into each conversation. There's no automatic loading. No tool integration. Just a markdown file you manually feed to the AI.

I have a spec.md for my current project that's 847 lines long. It has everything. Database schema. API endpoints. Business logic rules. UI component hierarchy. Testing requirements. It is comprehensive. It is also a pain to maintain and I have to paste it into every new ChatGPT conversation.

The manual nature is both a weakness and a strength. It works with every AI tool. But you pay for that portability with tedium.

CLAUDE.md vs .cursorrules vs llms.txt vs AGENTS.md: which one do you need?

It depends on which tools you use. CLAUDE.md is for Claude Code. .cursorrules is for Cursor. AGENTS.md aims to be universal but is still gaining adoption. llms.txt is for web-facing projects. Most developers using multiple AI tools end up maintaining several of these files.

CLAUDE.md

Claude Code only. Auto-loaded at session start. Supports path-scoped rules via .claude/rules/ directory. Most mature system with Skills for reusable workflows.

Tools: Claude Code

Format: Markdown with specific conventions

Scope: Project-wide and path-specific rules

.cursorrules / .cursor/rules/

Cursor only. Loaded in-editor automatically. Moving toward path-scoped rules similar to Claude. Works with any model Cursor supports.

Tools: Cursor

Format: Markdown instructions

Scope: Project-wide, evolving path-specific support

AGENTS.md

Tool-agnostic attempt at standardization. Recognized by GitHub Copilot, Windsurf, and growing list of tools. Still early but gaining momentum.

Tools: GitHub Copilot, Windsurf, others

Format: Markdown with flexible structure

Scope: Project-wide

llms.txt

Web-facing standard for LLM crawlers, not coding assistants. Helps AI understand your public project or API. Different purpose than development context files.

Tools: Perplexity, web crawlers

Format: Structured text sections

Scope: Public project documentation

spec.md

Manual convention, not a standard. Copy-pasted into conversations. No tool integration. Most portable but most tedious. Works everywhere because it is just text.

Tools: Any (manual paste)

Format: Freeform markdown

Scope: Whatever you write

If you use Claude Code, you need a CLAUDE.md. If you use Cursor, you need .cursorrules. If you use both, you need both. Nobody loves this, but that is the current state.

A question I keep seeing: "Should I just use AGENTS.md for everything?" Maybe eventually. But today, tool-specific files give you better integration and more features. CLAUDE.md with Skills is more powerful than generic AGENTS.md. The trade-off is maintenance burden versus functionality.

Where do manual context files break down?

Manual context files have three fundamental limitations. They are static and do not update when your code changes. They are platform-locked and only work with one tool. And they require manual maintenance that developers inevitably neglect as projects evolve.

The static problem: Your CLAUDE.md describes your project as it was when you wrote the file. Three weeks and forty commits later, the context is stale. You refactored the auth system but forgot to update the context file. The AI is working from outdated information and gives you subtly wrong suggestions. I've debugged "why is Claude suggesting the old pattern" too many times.

The platform-lock problem: CLAUDE.md does nothing in Cursor. .cursorrules does nothing in Claude Code. If you use multiple tools (and the JetBrains survey shows 73% of developers do), you are maintaining duplicate context files. Same information, different formats, constant synchronization burden.

The maintenance burden: Writing the initial file is easy. Keeping it accurate over months of development is a chore nobody prioritizes. The file drifts from reality. The AI drifts with it. Your suggestions get progressively worse as the gap widens.

These problems compound if you use AI tools multiple times per day across multiple providers. Which, if you are reading this, you probably do.

What does automated context management look like?

Automated context management means defining your project context once and having a tool keep it current as your codebase changes, then making that context available to any AI platform you use. Instead of maintaining separate static files for each tool, you maintain one source of truth that updates itself.

This is the problem Kept's Context Templating feature is designed to solve. Not as a sales pitch, but as an example of where the tooling is headed.

Here's how it works. You define your project context once through Kept's CLI. The CLI watches your project files and keeps the context template current. When you open any AI platform (ChatGPT, Claude, Gemini), the Kept Chrome extension can inject the relevant context. Smart compression ensures you are not wasting tokens on files that are not relevant to your current task.

Same context, every platform, automatically updated. Your CLAUDE.md describes behavior. Kept's context describes your actual code.

This feels like the natural evolution. CLAUDE.md was the right instinct. We recognized that AI tools need context to be useful. But maintaining that context manually across multiple tools is not sustainable. Automation is the obvious next step.

No. Your CLAUDE.md still works in Claude Code. Your .cursorrules still works in Cursor. Kept adds a layer that handles what these static files cannot: automatic updates when your code changes, portability across every AI platform, and smart compression to stay within token limits.

The bigger pattern: context is the new code

The explosion of context files signals something important. As AI tools become central to development workflows, the quality of context you provide determines the quality of output you get. Context engineering is becoming a core developer skill, not a hack or a workaround.

Martin Fowler's ThoughtWorks article calls it "the hidden productivity multiplier." Addy Osmani says "context packing is 80% of getting good AI output." These are not fringe ideas. This is mainstream thinking now.

The tools that figure out context management will win the next wave of AI development tooling. Claude Code's CLAUDE.md was a breakthrough because it eliminated the copy-paste tax. But it is still single-platform and static. The next breakthrough will be cross-platform and dynamic.

We are watching the ecosystem evolve in real time. Six months ago, nobody had heard of CLAUDE.md. Today it is in thousands of repos. Six months from now, automated context management will likely be standard.

The developers who figure out context management first will get dramatically better results from every AI tool they use. I have seen 10x differences in output quality between well-contextualized and poorly-contextualized AI sessions. Same model, same prompt, wildly different results based on context alone.

What happens next?

The manual context file era is necessary but temporary. We needed to discover that context files work before we could automate them. CLAUDE.md proved the value. Now we need to solve the maintenance and portability problems.

I expect three things to happen in 2026. First, more tools will adopt AGENTS.md or create a genuine standard for context files. Second, automated context management will become a standard feature, not a third-party add-on. Third, context will become more intelligent, understanding not just your files but your development patterns.

Kept is one approach to this problem. There will be others. The important thing is that we are moving past the copy-paste era. Context should follow you between tools. It should stay current as your code evolves. It should not require manual maintenance.

Your AI tools are only as good as the context you give them. Right now that context is scattered across half a dozen static files that go stale the moment you write them. It does not have to be that way.

The teams that solve context management will build the next generation of AI development tools. The developers who adopt these tools early will have a significant productivity advantage. This is not speculative. It is happening now.

I still maintain my CLAUDE.md and .cursorrules files. They work well for what they are. But I also run Kept's CLI to keep my context fresh across platforms. The combination gives me the best of both worlds: tool-specific behavioral rules and automatically updated project context.

This is where we are headed. Not replacing CLAUDE.md, but augmenting it. Not abandoning manual context files, but automating the tedious parts. The goal is simple: your AI tools should understand your project as well as you do, without you having to explain it every time.
